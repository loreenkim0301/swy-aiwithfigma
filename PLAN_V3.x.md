# V3.x 계획서 — Figma MCP 트랙

> **상태**: 설계 확정 (2026-03-27)
> **핵심 목표**: V2.x가 해결하지 못한 문제를 MCP로 개선한다

---

## 1. API vs MCP — 감각으로 이해하기

기술 용어 전에, 두 방식이 사람에게 어떻게 다른지부터.

### V2.x (Plugin API) — "자판기"

```
사용자가 할 수 있는 것: 버튼 누르기 (플러그인 실행)
결과: 항상 같은 것이 나온다
바꾸려면: 기계(코드) 안을 열어서 수정해야 한다
```

자판기는 정확하고 빠르지만, 메뉴가 코드에 고정되어 있다.
"아메리카노 말고 라떼로 바꿔줘" → 새 자판기 코드가 필요하다.

### V3.x (MCP) — "바리스타에게 말하기"

```
사용자가 할 수 있는 것: 원하는 것을 말하기
결과: AI(Claude)가 판단해서 Figma에 직접 반영
바꾸려면: 말로 다시 요청하면 된다
```

바리스타(Claude)는 주문을 이해하고, 조정하고, 직접 만든다.
코드를 알 필요가 없다.

---

## 2. V2.x의 진짜 문제 — 무엇이 불편했나

V2.x가 기술적으로 완성됐어도 실제 사용에서 남은 문제들:

| 문제 | 내용 |
|---|---|
| **사용자가 제한됨** | Figma Desktop + manifest.json 설치 필요. 개발자만 쓸 수 있음 |
| **디자인이 고정됨** | 블로그 레이아웃만 생성 가능. 다른 페이지를 원하면 코드를 다시 짜야 함 |
| **수정이 어려움** | "카드 간격을 좀 더 넓혀줘" → 코드를 열고 수치를 찾아 수정 → 재실행 |
| **단방향** | 플러그인이 만들고 닫힘. 대화가 없음. 피드백 루프 없음 |
| **느린 실험** | 새 디자인 아이디어를 시도하려면 개발자에게 요청 → 코드 수정 → 대기 |

## 3. V3.x가 개선하는 것

### 사용자가 달라진다

```
V2.x 사용자: 개발자 (코드를 쓸 수 있는 사람)
V3.x 사용자: 기획자, 디자이너 (말할 수 있는 사람)
```

### 플로우가 달라진다

**V2.x 플로우 (현재)**
```
기획자가 요구사항 전달
  → 개발자가 코드 수정 (수 시간 ~ 수 일)
    → 디자이너가 Figma에서 플러그인 실행
      → 결과 확인
        → 수정 필요 → 처음으로 돌아감
```

**V3.x 목표 플로우**
```
기획자/디자이너가 Claude에게 직접 요청 ("블로그 메인 페이지 만들어줘")
  → Claude가 HTML 생성 + Figma에 즉시 반영 (수 분)
    → 결과 확인
      → "카드 3개로 줄여줘" → Claude가 즉시 수정
        → 핸드오프 준비 완료
```

개발자가 중간에 없다. 기획자와 디자이너가 직접 왔다갔다할 수 있다.

### 성능이 달라진다

| 항목 | V2.x | V3.x 목표 |
|---|---|---|
| 첫 결과물까지 시간 | 개발자 코드 시간 포함 (수 시간~) | 프롬프트 입력 후 수 분 |
| 수정 사이클 | 코드 수정 → 재실행 | 말로 요청 → 즉시 반영 |
| 사용 가능한 사람 | 개발자 | 기획자, 디자이너 포함 누구나 |
| 만들 수 있는 디자인 | 블로그 레이아웃 1종 | 프롬프트로 설명 가능한 모든 것 |
| 시각 충실도 | 코드가 정의한 수준 | 브라우저 렌더링 100% |

---

## 4. 그러나 V3.x가 아직 풀지 못한 것

MCP로 넘어가면서 생기는 새로운 문제:

| V2.x에서 가능한 것 | V3.x에서 현재 불확실한 것 |
|---|---|
| Design Token 정밀 연결 (styleId 직접 지정) | MCP가 같은 수준으로 Token을 연결할 수 있는가? |
| Auto Layout 정밀 제어 (padding/gap 수치) | AI가 추론한 레이아웃이 V2.x만큼 정확한가? |
| 컴포넌트 생성 (createComponent) | MCP가 컴포넌트를 만들 수 있는가? |
| 결과 재현성 100% | AI 판단 개입 → 실행마다 다를 수 있음 |

**이것이 V3.x 실험의 핵심 질문이다.**
V3.x가 V2.x의 편의성을 가져오면서도 V2.x의 품질을 유지할 수 있는가?

---

## 5. 버전 정의

### V3.0 — 빠른 시각화: 프롬프트에서 Figma까지

**V2.x 대비 개선 목표**
- V2.x: 개발자가 있어야 실행 가능 → V3.0: 기획자가 혼자 실행 가능
- V2.x: 정해진 블로그 레이아웃만 → V3.0: 원하는 페이지를 프롬프트로 요청

**사용자 플로우**
```
1. 기획자가 Claude에게 원하는 페이지를 한국어로 설명
2. Claude가 HTML/CSS 작성
3. generate_figma_design으로 Figma에 즉시 캡처
4. 수 분 안에 시각 결과물 확보
```

**V2.0 대비 성능 목표**

| 항목 | V2.0 | V3.0 목표 |
|---|---|---|
| 실행 가능한 사람 | 개발자 | 기획자, 디자이너 |
| 첫 결과물까지 시간 | 플러그인 코드 필요 | 프롬프트 입력 후 수 분 |
| 시각 충실도 | 코드 정의 수준 | 브라우저 렌더링 100% |
| Design Token | ✅ | ❌ (V3.1에서 해결) |
| 핸드오프 가능 | 부분 | ❌ (V3.1에서 해결) |

**V3.0이 답해야 하는 질문**
- 기획자가 코드 없이 Figma에 시각 결과물을 만드는 경험이 실제로 작동하는가?
- 어느 정도의 프롬프트 퀄리티가 있어야 쓸 만한 결과물이 나오는가?

---

### V3.1 — 품질 보완: Design Token + Auto Layout 자동 적용

**V2.x 대비 개선 목표**
- V3.0 캡처 결과물(flat, 토큰 없음)에 V2.1 수준의 핸드오프 품질을 자동으로 부여
- 사람이 수동으로 정리하는 작업을 Claude가 대신

**사용자 플로우**
```
V3.0 결과물 (flat 레이어)
  → Claude가 get_design_context로 읽음
  → 색상·폰트를 분석해 어떤 Design Token인지 추론
  → use_figma로 Token 연결 + AL 그룹화 시도
  → 핸드오프 가능한 상태로 변환
```

**V2.1 대비 성능 목표**

| 항목 | V2.1 | V3.1 목표 |
|---|---|---|
| Design Token 연결 | ✅ 정확 (코드로 지정) | AI 추론 → 정확도 측정 필요 |
| Auto Layout | ✅ 정밀 (수치 직접 지정) | AI 판단 → 정밀도 측정 필요 |
| 수동 개입 없이 처리 | ✅ | ✅ (목표) |
| 사용 가능한 사람 | 개발자 | 기획자, 디자이너 |

**V3.1이 답해야 하는 질문**
- AI 추론 기반 Token 매핑이 실용적인 정확도(70% 이상)에 도달하는가?
- 도달하지 못한다면 → "MCP 캡처 + Plugin Mode B 하이브리드"를 공식 워크플로우로 정의

---

### V3.2 — 컴포넌트 자동 생성: V2.2와 동등한 수준 달성

**V2.x 대비 개선 목표**
- V2.2: 개발자가 어떤 요소가 컴포넌트인지 코드로 직접 지정
- V3.2: Claude가 반복 패턴을 보고 스스로 판단해서 컴포넌트로 변환

**사용자 플로우**
```
V3.1 결과물 (Token + AL 적용)
  → Claude가 레이어 구조 분석
  → 반복 패턴 감지 (Post Card, Button, Tag 등)
  → use_figma로 컴포넌트 변환 시도
  → Assets 패널 등록 확인
```

**V3.2이 답해야 하는 질문**
- MCP가 createComponent() 수준의 작업을 지원하는가?
- 지원한다면, AI가 반복 패턴을 얼마나 정확하게 감지하는가?
- 지원하지 않는다면 → V2.x Plugin + V3.x MCP의 역할 분담을 최종 정의

## 6. V3.x 전체가 끝났을 때 나와야 하는 결론

### 결론 A — MCP가 V2.x를 대체할 수 있다
```
조건: Token, AL, 컴포넌트 모두 MCP로 V2.x 수준 달성
의미: 개발자 없이 기획자·디자이너가 핸드오프까지 완성 가능
```

### 결론 B — MCP와 Plugin이 역할을 나눈다 (현재 예상)
```
MCP: 아이디에이션 → 빠른 시각화 → 방향 검토
Plugin API: 수렴 → 정밀 구조화 → 핸드오프

워크플로우:
기획자 (MCP로 빠른 시각화)
  → 방향 확정
    → 디자이너 (Plugin으로 핸드오프 품질 완성)
      → 개발자 핸드오프
```

### 결론 C — MCP의 현재 한계가 명확하다
```
한계 확인 자체가 연구 결과
어느 기능이 MCP로 불가능한지 데이터로 기록
향후 Figma MCP 업데이트 시 재검증 기준이 됨
```
